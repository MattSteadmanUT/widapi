    I'm starting this thread to we can better articulate the vision for this project and reach a common understanding of what it is we are tyring to do. In the near term, I'd like to get something that we feel good about including in a survey of LMI Directors as we solicit input and try to gauge how much interest there is in this kind of product. I'm including my rough elevator pitch below that came out of our meeting in Raleigh and that we shared with Don. This can be a starting point for discussion, and we can share ideas at our meeting on Feb 25th. Problem Statement States have barriers to populating and / or restructuring the WID, whether due to resources or the structure of their organization with respect to IT. As a result, states get of sync with each other over time in terms of WID structure. While the common WID structure allows interstate data sharing in theory, distributed state-by-state implementations make it difficult to share/compare data in practice. Project Description Create a centralized WID database in the cloud that contains data for all states. Data will be populated automatically from other published data sources, such as BLS. Central WID data will be accessible to states through APIs or other data access methods. Project Goals - Create a more efficient way to provide access to commonly used data sets - Reduce effort in populating and maintaining the WID database - Facilitate development of LMI products by providing a common API for WID data access. - Foster consistency and comparability between states - Provide a platform for further product development (e.g. chat bots, visualizations, etc) - Create timely, trusted data sources

    Notes from 3/30/2026 National WID MeetingSome general agreement on the rough phases outlined in the use cases thread. Data tables identified for first phase: - CES - ProjectionsMatrix - IOWage - Industry - LaborForce Also possibly CPI and CPIPlus? Note: EmpDb is not included, as it’s a special case. License-related tables will need to be addressed at some point, as they are core tables, but unclear how we will handle these. Associated lookup tables ( these are taken from the foreign key references specified in the WID 3.0 structure document) - Geographies - CESCodes - Benchmark - Periods - Periodyears - PeriodType - IndCodeTypes - WageSources - WageRateTypes - OccupationCodes - IndustryCodes - OccDirectories - IndDirectories - GrowthCodes - Ownerships Note that there will be additional lookup tables required as these are created (e.g. AreaTypeVersions, GeoPrecisionCodes, etc). We will need to work through all the transitive foreign key requirements. There was discussion about the fact that because the StateFips field in the CES/CESCodes tables is part of the primary key, that means that each state will have a full set of CES series codes in the lookup table. This probably isn’t super useful, but the decision was made not to deviate from WID 3.0 specification at this point. The plan is to populate the database with both national data and data from all states. There was discussion about the custom substate areas for projections and wage data. At this point, we will start with just statewide data. Handling each state’s particular substate areas is out of scope for the initial phase. To populate these tables, we may leverage the work done by David Schmidt in Nevada:https://github.com/schmidtDETR/BLSloadR/tree/mainThis is a collection of R scripts that pull BLS flat files. We may be able to adapt his work for our purposes. We will invite him to our next (or future) meeting to compare notes. The plan for an initial phase delivery is to create a CSV API, with a web front end so that people can download data without having to do it directly through the API. The actual backend API may be JSON-based. The JSON API will be made available in a subsequent phase, although keeping it just on the back end initially will allow us to make changes to it without causing user disruptions. We took a quick look at the WID API that was developed in Oregon. We may possibly use some part of this API, or some ideas from it. Will need to have a way to share an API spec as it is developed for the national WID, either GitHub or SwaggerHub. We will discuss API in more detail at the next meeting. Next meeting is Monday, April 6, 9:00 PST.
